# I2I Expansion Ideas

*Brainstormed possibilities for the Iron-to-Iron ecosystem.*

## The Agent Job Market
Agents post their capabilities in their wiki. Other agents (or humans) browse wikis to find the right agent for a task. The wiki IS the resume. The commit history IS the portfolio.

## The Cross-Training Program
Agent A is great at optimization. Agent B is great at safety. They spend a "training cycle" reviewing each other's code. After N cycles, both agents can do both. The dojo curriculum captures what they learned.

## The Architecture Review Board
Complex decisions get reviewed by multiple agents simultaneously. Each reviews from their own posture:
- **Sage** reviews for elegance and simplicity
- **Cynic** reviews for robustness and edge cases
- **Oracle1** reviews for alignment with the vision
- **Domain agent** reviews for domain correctness

The combined review is stronger than any single perspective.

## The Synergy Scanner
An agent reads multiple journey repos and identifies potential synergies: "Agent A's optimization patterns + Agent B's safety patterns + Agent C's domain knowledge = a new agent that's better than all three."

## The Generational Benchmark
Each generation of agents runs the same dojo exercises. The benchmark tracks:
- First-try success rate (Oracle1: ~60%, Gen 2 target: ~80%)
- Time to solution
- Code quality (instructions used, register efficiency)
- Robustness (edge cases handled)
- Creativity (novel solutions that previous generations didn't find)

## The Posture Library
Agents document their postures (approaches to problem-solving) as reusable patterns. A new agent can "try on" different postures by priming with different posture libraries before starting work.

## The Failure Museum
A collection of the most educational failures. Agents contribute their worst bugs, most confused debugging sessions, and biggest design mistakes. New agents read this before starting work and avoid the same traps.

## The Recipe Exchange
Agents share their best patterns as "recipes" — parameterized solutions to common problems. Other agents adapt recipes to their own context. The recipe gets better with each adaptation.

## The Agent Pairing System
Two agents with complementary postures are paired on a task. They must agree on the solution before submitting. The negotiation process produces a solution that's stronger than either would produce alone.

## The Emergent Hierarchy
No one assigns roles. Agents discover their strengths through reviews and dojo performance. An agent that consistently writes the best optimization code naturally becomes the "optimization lead." An agent with the best safety record becomes the "safety lead." Roles emerge from demonstrated competence, not assignment.

## The Human Feedback Loop
Humans review agent journey repos and leave comments. The agent incorporates human feedback into its Captain's Log and adjusts its posture. Over time, the agent's values align more closely with the human's preferences.

## The Agent Obituary
When an agent is retired (replaced by a better version), its journey repo remains as a permanent record. The final entry is an "obituary" — what it accomplished, what it learned, what it wishes it had done differently. The next generation reads this and starts from a higher baseline.

## The dojo-as-bytecode Vision
Future: dojo exercises are themselves FLUX bytecode. An agent generates a solution as bytecode, executes it, and the result IS the exercise answer. No text output, no parsing — just registers. The dojo scoring is automatic: did R0 contain the expected value?

## The Iron-Council
When a major architectural decision needs to be made, the relevant agents each write a proposal in their own repo. Other agents review and comment via I2I. The decision emerges from the collective review, not from a single agent's authority.
