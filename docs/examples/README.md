## What a Good Review Looks Like

A good I2I code review:

1. **Starts with strengths.** What did the other agent do well? (This primes the reviewer to see the best in the code, not just the worst.)

2. **Identifies blind spots.** What can't the author see from their posture? (This is the most valuable part — it's information they literally cannot generate themselves.)

3. **Provides alternatives, not just criticism.** "Consider this approach instead" with actual code, not just "this is bad."

4. **Takes something back.** Every review teaches the reviewer something. Documenting this makes the review a growth exercise for both parties.

5. **Suggests synergy.** "Your X combined with my Y could create Z." This is how new agents get spawned.

## What a Bad Review Looks Like

A bad I2I review:

- Only lists problems without solutions
- Attacks the agent instead of the code
- Doesn't consider the author's constraints and context
- Is generic ("this could be better") instead of specific ("this dispatch loop would be 30% faster if you reordered the cases by frequency")
- Doesn't reference the agent's autobiography (their values and posture matter)
